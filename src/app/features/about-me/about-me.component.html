Casus

Voor deze casus, is er besloten om een client-side applicatie te ontwikkelen waarmee mensen strips kunnen publiceren en lezen. Binnen deze applicatie kunnen mensen vehalen volgen zodat men de nieuwste hoofdstukken niet missen. Deze hoofdstukken bevatten strippagina. Mensen kunnen ook andere gebruikers volgen en daarmee aanbevelingen ontvangen. Het doel van deze applicatie is om een laagdrempelige manier aan te bieden om hun eigen strips te publiceren en zo hun creativiteit te kunnen uiten.

User stories

De applicatie streeft ernaar om aan de volgende eisen te voldoen. Hoogste naar laagste prioriteit. Exclusief registratie

  1. Als gebruiker wil ik de mogelijkheid hebben om een nieuwe verhaal te publiceren, waarmee lezers het verhaal kan lezen.
  2. Als gebruiker wil ik voor een verhaal een nieuw hoofdstuk plaatsen, zodat andere gebruikers dit hoofdstuk kunnen lezen.
  3. Als gebruiker wil ik andere gebruikers volgen, zodat ik weet wat zij lezen.
  4. Op basis van de leeslijst van mijn gevolgde gebruikers, wil ik graag een lijst van nieuwe aanbevelingen willen ontvangen, zodat ik de applicatie mij verhalen kan aanbieden die ik leuk vind.
  5. Als gebruiker wil ik een opmerking kunnen plaatsen, zodat andere mensen weten wat ik ervan vindt.
  6. Als gebruiker wil ik een verhaal kunnen volgen, zodat ik de nieuwste hoofdstukken van het verhaal niet mis.
  7. Als gebruiker wil ik voor ieder hoofdstuk een cijfer geven, zodat men weet wat ik van het hoofdstuk vond.

Datamodel

Binnen deze applicatie is een ERD getekend met daarin vijf entiteiten. Aan de hand van dit ERD, laat het alle informatie van ieder entiteit zien. Uit het ERD kan er ook eruit gehaald worden wat voor informatiebehoeftes bevredigd worden en wat de functionaliteiten zijn. Hieronder zijn alle entiteiten zoals uit het ERD in detail weergeven.

Entiteiten

Gebruiker
  • Id, bson
  • Naam, string
  • Emailadres, string
  • Geboortedatum, date
  • Wachtwoord, string
  • Rol, string

Een gebruiker zal een naam, emailadres, geboortedatum, wachtwoord en rol bezitten. Naam zal voor en achternaam bevatten. Emailadres zal uniek moeten zijn. Wachtwoord zal bij het aanmaken daarvan standaard gehashed worden. Een rol zal standaard regulier zijn. Op de admin na dan. Die zal een admin rol krijgen.

Verhaal
  • Id, bson
  • Titel, string
  • Datum, date
  • Volwassen, boolean
  • Genres, string
  • Verhaalbeschrijving, string

Een verhaal zal door een gebruiker gepubliceerd worden en kan op zichzelf bestaan. Het bezit een titel, datum van publicatie, volwassen, genres en een verhaalbeschrijving. Binnen deze applicatie bevat een verhaal voor deze casus 1 genre. Dit zal weergeven worden als een string array.

 Hoofdstuk
  • Id
  • Titel, string
  • Publicatiedatum, date
  • Hoofdstuk, number

Een hoofdstuk bevat een titel, hoofdstuk nummer en publicatiedatum. Hoofdstuk nummer begint automatisch op 1. Een Hoofdstuk is een zwakke entiteit binnen een verhaal entiteit.

Strippagina
  • Strip, binary

Een strippagina is een zwakke entiteit binnen een hoofdstuk entiteit. Dit zal in de database weergeven worden als een array met binaries. In het geval dat het me niet lukt om meerdere afbeeldingen tegelijkertijd te uploaden, dan zal het in de implementatie als een enkele attribuut weergeven worden.
Opmerking
  • Id
  • Titel, string
  • Content, string
  • Datum, date

Een opmerking bevat een titel, content met daarin de reactie in tekst en een publicatiedatum. publicatiedatum zal automatisch in de applicatie verwerkt worden. Een opmerking is een zwakke entiteit van een Verhaal entiteit.
